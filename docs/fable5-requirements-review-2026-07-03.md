## 2. 価値仮説の再定義（Codex相談反映済み）

### 旧仮説（過大だったもの）
「整形済みテキスト＋魚拓導線がスクショ・手動コピペの**代わり**になる」→ **棄却**。証拠保全実務の主役はスクショ+PDFであり、gyo.tc魚拓はXログイン壁で成否不安定（§4）。テキストが主役という主張は実務と乖離。

### 改訂版 価値仮説
> スクショ / PDF / 魚拓取得の**補助**として、X投稿の引用・相談・記録に必要なメタデータと本文を、欠落なく・定型で・1操作で揃える「**記録補助ツール**」。

### 利用者セグメント（優先順位を改訂）
1. **D. 自分・運営者自身**（主セグメント。個人ツールとしての常用が最初の成功条件）
2. **B. ブログ・note・Obsidian等でX投稿を引用・記録したい書き手**
3. A. トラブル相談前の**一次整理補助**をしたい個人（「法的証拠保全」ではない。スクショ/PDF併用を案内する前提）
4. C. 小規模な研究・報道**メモ**（機関向け・大量収集・監視・削除対策・第三者向け保全業務はMVP外として明示的に切る）

## 4. 市場・仕様調査メモ（2026-07-03 Web調査、Sonnet agent実施）

確認日はすべて 2026-07-03。詳細な出典は各項目に記載。

1. **X API v2**: 2026-02-06 に料金体系が従量課金中心へ転換。**Free tier新規提供は廃止**（既存Freeは$10クレジット付きで従量へ移行）。post read **$0.005/件**、月200万readキャップ。`GET /2/tweets/:id` は App 450req/15分。（出典: docs.x.com/x-api/getting-started/pricing ほか。single lookupの$0.005区分該当は二次ソース一致による推定）
   → **BYOTの敷居が上昇。oEmbed fallbackが実質的な主経路になる**。
2. **公式oEmbed** (`publish.x.com/oembed`): 稼働中（HTTP 200 実測）。無認証・無料。`cache_age` 実測 約36500日。rate limit の公式数値なし（コミュニティ報告 180req/15分説）。廃止アナウンスなし、ただし2024-2025年に間欠的404報告 → fallback設計は引き続き妥当。
3. **gyo.tc / megalodon.jp**: 両方稼働中（megalodon に 2026-07-02 の新規魚拓を確認）。ただし2023年のXログイン壁以降「X投稿の魚拓が取れない」報告が継続。完全不可の断定情報はなし。
4. **archive.today**: ログイン壁回避でX保存可との評価がある一方、2026-01 に CAPTCHA 改変による DDoS 加担が発覚し、2026-02 に英語版 Wikipedia がリンクをブラックリスト化。信頼性に重大な懸念。**Wayback Machine** は X 投稿の表示エラー報告が継続、解消未確認。
5. **競合（日本語圏）**: Twitter魚拓（twtr.satoru.net、画像化保存）、ウェブ魚拓（gyo.tc）、Twilog（自分のポストのログ）、Kiroku 等。テキスト+メタデータ定型抽出に特化した直接競合は今回の調査では見つからず（未確認領域あり）。
6. **弁護士推奨の証拠保全実務**: 絶対表記の日時・固有URL・表示名/@ID・本文・文脈を**スクショ（PC推奨）+PDF**で保存が主流。プロバイダのログ保存は3〜6か月で時間との勝負。→ 本MVPの出力項目は推奨項目とほぼ一致するが、**実務上テキストは補助**。
7. **X規約**: オフライン保存コンテンツには削除同期義務（削除要請から24時間以内）。スクレイピング禁止。本文非保存・非HTML描画・出典リンク必須の現設計は規約リスクを最小化しており維持が安全。

## 5. UI再設計方針（改訂: 公開範囲決定後に着手）

- 現行UIは C-01〜C-13 で機能整備済み。**視覚再設計は Q1（公開範囲）の回答後**に着手する（自分専用なら投資最小、一般公開なら frontend-design skill でブランド確立）。
- デザイン方針として「証拠保全ツールらしい信頼感」は**採用しない**（位置づけと乖離するため）。採用するなら「記録補助ツールとしての実直さ・読みやすさ」。
- **運用者向け判断画面は不採用**（Codex相談一致: Markdownの決定票で足りる。低ROI・ゲート④案件）。
- 着手時の進め方: frontend-design skill で方針（配色4-6 hex / 書体 / レイアウト / signature）→ wireframe → オーナー確認 → 実装は codex-deep / Sonnet 5 へ委譲。
